# DEVELOPERS.md — working on docusearch

Audience: a human maintainer or an AI agent about to change this code. Everything below was
read out of `src/docusearch/` at the time of writing; where a claim could not be verified from
code it is called out under **Unverified**.

Read this before editing. The most valuable section is [The seams](#2-the-seams--where-to-extend-by-intent):
most changes belong at an existing extension point, and designing around one is how this
codebase gets worse.

---

## 1. Orientation

docusearch turns folders of documents, source code, and test data into a SQLite catalog
(FTS5 BM25 + optional vectors) and serves it over a CLI, a REST API, and MCP. Reports are
generated by code from a spec, never freehand by a model, and every claim carries a citation.

### Module map

| Module | Purpose (one line) |
|---|---|
| `__init__.py` | The public API surface: `Catalog`, `Config`, `serve`, `__version__`. `serve` is lazy so `import docusearch` never pulls FastAPI. |
| `__main__.py` | `python -m docusearch` → `cli.main`. |
| `_version.py` | Single source of the version string. |
| **`store.py`** | The *only* module that touches SQLite: schema, migrations, every read and write. |
| **`ingest.py`** | Discover → extract → chunk → link/image → index. Largest module by design. |
| **`search.py`** | BM25 (FTS5), vector ANN (hnswlib, numpy fallback), hybrid RRF fusion, role filtering. |
| **`server.py`** | FastAPI REST + MCP-over-HTTP. Both surfaces are thin wrappers over one `Service` class. |
| **`config.py`** | Schema-driven YAML config: one `SCHEMA` list generates the template, the defaults, and the validator. |
| **`report.py`** | Renders a cited report to `md` / `html` / `html-slide`; owns the four themes. |
| **`report_export.py`** | Renders the same report to `pdf` / `docx` / `pptx` / `xlsx`, returning bytes. |

| `wafer.py` | Wafer map, mother-lot, and long-term yield-trend views over parsed STDF parts. |
| `mcp_client.py` | Tiny synchronous MCP client — the transport behind `docusearch stdf`/`data`/`code`. |

### Load-bearing modules

`store.py`, `server.py`, `ingest.py`, `search.py`, `report.py`, `report_export.py`,
`config.py`, `profiles.py`, `igxl_rules.py`. Changes here have blast radius across every
other module; changes elsewhere generally do not.

Two structural rules the codebase enforces on itself:

- **One implementation per concept.** SQL lives only in `store.py`. Citation parsing lives
  only in `citations.py`. There is no `utils.py` and one must not be created.
- **Heavy imports are lazy.** `fastapi`, `torch`, `pymupdf`, `python-docx`, `playwright`,
  `tree_sitter`, `anthropic`, `matplotlib`, `plotly` are all imported inside the function
  that needs them, so a BM25-only standalone user never loads them. Keep it that way when
  adding a dependency — see the extras in `pyproject.toml`, each of which carries the
  justification for why it exists.

## 2. The seams — where to extend, by intent

Find your intent in the left column; change only what the right column names.

| I want to… | Go to | Do not |
|---|---|---|
| Support a new file format | `ingest.extract_document` + a new `extract_<fmt>` | Touch the chunker, store, or search |
| Recognise structure in one document *family* | `profiles.py` | Fork a parser |
| Add a config key | `config.SCHEMA` | Add it to a dataclass only |
| Store a new fact | `store._SCHEMA_Vn` + `_MIGRATIONS` + `SCHEMA_VERSION` | Edit a shipped migration |
| Expose a new agent capability | `server.py` `@mcp.tool()` | Put logic in the tool body |
| Add a report output format | `report.py` (text) or `report_export.py` (binary) | Bypass the citation guard |

### 2.1 New file format → `ingest.extract_document`

```python
def extract_document(path, ext, *, content_selector="", strip_selectors=()) -> ExtractedDoc:
```

Dispatch is by extension (`pdf`, `docx`, `md`/`markdown`, `pptx`, `xlsx`, everything else
falls through to HTML). Every extractor returns an `ExtractedDoc` — `title`, `segments`,
`links`, `images` — and **the rest of the pipeline is format-agnostic** because of it:
`chunk_document`, link/relation mining, image retention, FTS indexing and embedding all
operate on `ExtractedDoc`/`Chunk` and have never heard of PDF.

**Why the seam is here.** One parser per format, one chunker for all of them. If a format
gets its own chunker you now have N chunkers to keep in step with `chunk_tokens`, overlap,
and the "a code segment is never split" rule — and the format that gets fixed last silently
retrieves worse. Adding a format is: one `extract_<fmt>(data: bytes) -> ExtractedDoc`, one
branch here, one lazily-imported optional extra in `pyproject.toml`.

### 2.2 New document family / enrichment → `profiles.py`

A `Profile` runs **after** the extension-chosen parser and only **adds** fields it
recognises:

```python
class Profile:
    name = "generic"
    def facts(self, title: str, chunk_texts: Sequence[str]) -> DocFacts: ...

PROFILES: dict[str, Profile] = {p.name: p for p in (IgxlProfile(),)}
```

Register your profile in `PROFILES`, then set `profile: <name>` on the source in
`docusearch.yaml` (`SourceConfig.profile`, default `""`). `profiles.get()` returns `None`
for blank or unknown names, so an unrecognised profile is a silent no-op, not an error.
`ingest` calls it at one place only (`prof.facts(...)`, around the `store.add_document`
call) and writes the result onto the document row.

**Why additive rather than a parser fork.** The IG-XL help set is HTML. If "IG-XL" became
its own parser, every improvement to generic HTML extraction — a boilerplate fix, a table
handling change, a heading-path bug — would have to be ported into it or the family would
rot. Instead the family shares the one HTML implementation and layers on the three things a
generic parser cannot know are meaningful: which **instrument** a page governs, which
**release** its rules start at, and whether a passage states a **limit**. A profile that
recognises nothing leaves ingest byte-identical to generic behaviour, which is what makes
adding one safe.

The `IgxlProfile` also shows the discipline expected of a profile: instrument names come
from a **closed whitelist**, not "any word after `Instrument:`" — the open version pulled in
`Support` and `HPM` from headings that ran into the label. Regexes are deliberately tight
("cannot be merged" is a rule; "you cannot see the grid" is prose).

### 2.3 New config field → `config.py` (schema-driven — and a real trap)

`config.SCHEMA` is a tuple of `_Field` / `_Section` / `_ListSection` records. That one list
generates **all three** of:

1. the fully-commented YAML that `docusearch init` writes (`render_template`),
2. the default mapping every load starts from (`_default_mapping`),
3. the validator, including closed-enum checking (`_validate`).

**The trap.** `_merged()` walks `SCHEMA` and copies only keys that appear in it:

```python
for node in SCHEMA:
    if node.key not in raw:
        continue
```

A key present in your YAML but absent from `SCHEMA` never reaches `Config._from_mapping` —
it is dropped, with only a `UserWarning: unknown config key '<key>' (ignored)`. Adding a
field to the `Config` dataclass, or reading `raw[...]` somewhere else, does not help: the
value was already discarded upstream. **This has actually bitten us.** The fix is always the
same: add a `_Field` to `SCHEMA` first, then thread it through `_from_mapping` into the
typed dataclass.

Corollary: because the template is generated, never hand-edit `docusearch.example.yaml` to
document a new key — regenerate it. Docs, defaults, and validation cannot drift apart by
construction, and that property is the whole point of the design.

### 2.4 New schema column → `store.py`

```python
SCHEMA_VERSION = 11
_MIGRATIONS: tuple[tuple[int, str], ...] = ((1, _SCHEMA_V1), ..., (11, _SCHEMA_V11))
```

Add `_SCHEMA_V12`, append `(12, _SCHEMA_V12)` to `_MIGRATIONS`, bump `SCHEMA_VERSION`.
Migrations are **append-only** — an already-shipped migration is never edited, because
existing databases have already run it. `Store.open` refuses a database whose recorded
version is newer than `SCHEMA_VERSION` with an actionable message rather than corrupting it.

**The `documents` vs `chunks` convention.** Doc-level facts live on `documents`; all chunks
inherit them by membership. Only genuinely per-passage facts live on `chunks`. Migration 10
states the reasoning in the code:

> A statement like "Instrument: UltraPin1600" or "V11.00.00 and later" governs the whole
> page, so it belongs on the document: every one of that page's chunks inherits it by
> membership. Storing it per-chunk would duplicate it N times and let the two drift.
> `restricts` is the one genuinely per-chunk signal.

Migration 11 shows the other half of the rule: a fact that is a property of *content*, not
of a document row, gets its own table. `images` rows are owned by ingest and deleted on
re-ingest, which silently destroyed every agent-written image description — so
`image_vision` is keyed by `sha256` (the content identity) and ingest never writes it.
Before adding a column, ask which of the three it is: document-scoped, passage-scoped, or
content-scoped.

### 2.5 New MCP tool → `server.py`

Tools are declared with `@mcp.tool()` inside the server's tool-registration block (currently
~30 tools: `help`, `search_docs`, `get_document`, `related_documents`, `catalog_stats`,
`build_report`, `report_format`, the `stdf_*` / `wafer_map` / `data_*` / `code_*` families,
`ingest_docs`, `submit_feedback`, `get_image`, `describe_image`, …).

Three rules:

- **The tool body is a wrapper.** All logic lives on `Service`; REST routes and MCP tools
  call the same method so the two surfaces cannot diverge. Feature parity between CLI and
  MCP is a standing gate requirement, and this is what makes it cheap.
- **The docstring is the agent-facing contract.** It is the entire specification the model
  sees. Descriptions are kept to roughly one line so that merely connecting the server costs
  almost no context; an agent that actually needs detail calls `help()`, which returns the
  full workflow on demand.
- **Images use `mcp.server.fastmcp.Image`**, not base64 in a JSON field:
  `from mcp.server.fastmcp import Image as _McpImage` … `_McpImage(path=str(path))`
  (see `get_image`). Putting binary through a JSON reply is exactly the context cost the
  compact search payload exists to avoid.

### 2.6 New report format → `report.py` vs `report_export.py`

| | `report.py` | `report_export.py` |
|---|---|---|
| Formats | `md`, `html`, `html-slide` | `pdf`, `docx`, `pptx`, `xlsx` (`EXPORT_FORMATS`) |
| Returns | `str` | `bytes` |
| Entry point | `render_report(...)` | `export_report(...)` |
| Themes | `THEMES` = `midnight`, `paper`, `slate`, `contrast` | PDF inherits them via the HTML |

`render_report` verifies the **whole surface** — title, subtitle, request, requested_by,
model, classification, every heading, every section body, audience, sources — against the
evidence set and raises `citations.CitationError` before rendering a byte. `export_report`
runs the same guard on its own path. Both must, or a format becomes a hole in the
invariant.

PDF is produced by printing the HTML report in headless chromium (playwright), deliberately:
one layout to maintain instead of two, and the page's JS runs so interactive plotly charts
actually appear in the PDF. `report_export.guidance(fmt)` is the single place stating what
good content looks like per format — the MCP tool, `help()`, and the skill all read from it,
because a section written as four dense paragraphs becomes four dense bullets on a slide and
the *author* has to know the target before writing.

`Service.build_report_file(spec, base_url=..., fmt=...)` is the normal entry point for
server-side rendering: it renders, writes the file under `tmp_dir/reports/` via
`report_store.write`, sweeps expired reports per `reports.retain_days`, and returns
`{fmt, filename, url, path, bytes}` — plus an inline `report` string for `md`/`html` only.
The binary formats deliberately do not come back inline.

Binary writers also sanitize: `xml_safe()` strips control characters OOXML rejects, and
`xlsx_cell()` prefixes a leading `= + - @` with `'` so a cell stays text and never becomes a
live formula in whoever opens the file (CWE-1236). Any new cell-writing path must use them.

## 3. Invariants that must not break

**1. Every claim carries `[D:<doc_id>#<chunk_id>]` or `[GK]`, and `citations.verify` refuses
anything outside the evidence set — on every format.**
`verify()` checks the **pair**, not just the chunk id: chunk ids are globally unique, so
`[D:1#2]` where chunk 2 belongs to another document is a mis-attribution that would resolve
to the wrong source URL. `[GK]` is never a violation. Both `render_report` and
`export_report` call it before writing. *Why:* this is the only mechanism preventing a
fabricated reference, and it only works if it holds on **all seven** formats — a pptx path
that skips it is a hallucination channel that looks like a report.

**2. A citation must resolve to a real source document in what the reader sees, never a bare
id.** `report.reference_targets(db_path, evidence, base_url=...)` maps each cited pair to
`(href, "store — title — heading")` via `store.citation_target`. The label is identical
everywhere; only the href differs by context — a served report links to
`/v1/documents/<id>?chunk=<n>`, the local CLI links to the original `file://` document.
*Why:* a reader must be able to click through and check the claim. Note the known gap in
`BACKLOG.md` §4: inline `[D:…]` ids still leak into report bodies in some paths; the fix
belongs in the renderer, not in a prompt (prompt-level pressure has failed three times).

**3. Scope must never be flattened into a universal claim.** An instrument- or
version-specific statement recorded or reported as universal is a wrong answer stated
confidently — the failure the whole profile/scope machinery exists to prevent. `instrument`
and `applies_from` live on `documents`; `instrument` is a filter on `Service.search`,
`search_docs`, and `/v1/search`. In `igxl_rules`, the even/odd channel rule applies **only**
to three-bit scan types — applying it to two-bit chains invents a constraint, skipping it on
three-bit chains misses a real defect. Both directions are wrong answers.

**4. Determinism.** Ranking is deterministic by construction: identical index + query ⇒
identical ranked results, tie-broken on `(doc_id, chunk_id)`. Anything model-driven in
validation runs at temperature 0 with recorded seeds. Plot rendering uses matplotlib Agg for
deterministic PNG data-URIs. *Why:* a gate's double-run byte-identical check is the only
cheap proof that a retrieval change did what you think it did.

> **Verified defect:** `vision.AnthropicProvider` calls `messages.create(model=…,
> max_tokens=…, output_config=…, messages=[…])` with **no `temperature`
> parameter** (`src/docusearch/vision.py` ~line 265), while `enrich.py`'s docstring says it
> "reuses the temperature-0 Claude backend from `vision.py`". Either the call should pass
> `temperature=0` or the docstring is wrong. Flagged, not fixed — this file was out of scope.

**5. All generated output lands under the config's `tmp_dir`.** Reports go to
`tmp_dir/reports/`, logs to `tmp_dir/logs/<date>.jsonl`. `.gitignore` excludes `tmp/`,
`staging/`, `*.db`, `*.hnsw`. *Why:* the repo stays a clean deployable, and an operator can
delete one directory to reclaim everything generated.

**6. No corpus-specific tokens, paths, or answers in `src/`.** Closed vocabularies that
belong to a *document family* live behind a named `Profile` or an explicitly-scoped rules
module (`igxl_rules`), driven by config — never sprinkled through the pipeline. The red team
greps for this.

## 4. Working practices

### Before every commit

```bash
ruff check src/
ruff format src/
mypy            # config in pyproject: strict, packages = ["docusearch"], python_version 3.12
```

`ruff` is pinned at `0.14.0` and `mypy` at `1.20.0` in the `[dev]` extra. Lint selects
`E,F,I,UP,B,SIM,PTH,W`; `PTH` is on deliberately — Windows 11 is a first-class target, so
use `pathlib`, spawn-safe multiprocessing, and no POSIX-only calls. mypy is `strict` with
`warn_unreachable`; the only relaxation is `disallow_untyped_decorators = false` scoped to
`docusearch.server`, because FastMCP's `@mcp.tool()` cannot be typed.

### Tests are not in this repo

`[tool.pytest.ini_options] testpaths = ["tests"]` points at a `tests/` directory that lives
in the **parent workspace** (`/Users/speck/Downloads/docusearch-kickoff/`), alongside
`harness/`, `redteam/`, `corpora/`, `scripts/`, `WORKLOG.md`, `CHANGELOG.md`, and the
architecture contract. Run `pytest` from the parent, not from here. Coverage gate is
`fail_under = 85`.

### Running the server locally

```bash
./start-server.sh                                   # background, binds 0.0.0.0:8321, pidfile + log
docusearch serve --config docusearch.yaml --host 127.0.0.1 --port 8321   # foreground
```

`start-server.sh` reads `DOCUSEARCH_CONFIG` / `DOCUSEARCH_HOST` / `DOCUSEARCH_PORT`, refuses
to stomp a server it did not start, logs to `tmp/server.log`, and prints the MCP URL
(`http://host:port/mcp`) and REST docs URL. `.mcp.json` in this repo already points a local
Claude Code at `http://localhost:8321/mcp`.

> **Trap: editing `report.py` or `server.py` requires a server restart.** The running
> uvicorn process holds the imported modules; a running server will happily keep serving the
> old renderer. Stale servers have produced genuinely confusing results — a "fix" that
> appears not to work, or a report that still shows the bug you just removed. Restart
> (`./start-server.sh` answers `y` to restart) before concluding anything about a report or
> tool change.

### Commit discipline

TDD: failing test → minimal code → refactor. Minimal means the least code that turns the
test green — no speculative abstractions, and delete dead code. Commit messages cite
requirement ids (`R-XXX-n`) from the architecture contract in the parent workspace.

## 5. Performance notes that matter

Measured numbers in `README.md` (14,938 docs / 126,469 chunks, `bge-small-en-v1.5`,
Apple-Silicon CPU single thread): BM25 p95 **13 ms**, hybrid p95 **24 ms**, HNSW alone
0.1 ms, federated fan-out over 3 stores ~15 ms.

Practically, an agent-facing call is **~0.1–0.3 s for a search and ~0.01–0.6 s to render a
report**. So end-to-end latency in an agent session is dominated by **model inference and
the number of sequential tool round-trips**, not by docusearch. The optimisation that pays
is batching: `search_docs` takes a *list* of queries and should always be sent all
phrasings at once, never one per call. Optimising the SQL further generally is not.

**The `search_docs` reply shape is deliberately compact and must stay that way.** From the
comment in `server.py`:

- a flat list of self-describing hits cost ~0.65 KB each, and a normal batched search
  (8 queries × top_k 10) overran the client's tool-output cap at 52 KB — which spilled the
  reply to a temp file and burned the agent's run parsing it;
- the cost was duplication, not content: 80 hits spanned 16 documents.

Three collapses fix it with nothing truncated: per-document `title`/`path`/`fmt` stated once
in a shared `documents` map; hits emitted as rows under a declared `hit_fields` header so no
key name repeats; anything derivable (URL prefix, embed model, search mode) stated once.
`cite` leads every row rather than a `(doc_id, chunk_id)` pair, because it is the exact
string the model must emit and so cannot be mis-paired the way two positional integers can.
`score` and `images` are dropped — rows are already ranked, and search never populated
`images`. Wide batches self-limit: above `_MCP_BATCH_QUERIES = 4` queries, `top_k` clamps to
`_MCP_BATCH_TOP_K = 5`.

If you add a field to a hit row, add it to `hit_fields` and account for the per-hit cost ×
top_k × queries. Regressing to a self-describing shape is a known-bad direction (`BACKLOG.md`
§4 tracks it).

Other performance facts worth knowing:

- The embedding model and vector index load **once** and stay warm in the server; SQLite is
  opened per request (WAL, thread-safe).
- Ingest is incremental — unchanged files are skipped by content hash. `--force` re-parses,
  `--reembed` only rebuilds vectors (much cheaper).
- STDF per-test values come from SQL, not materialised run objects: building them as objects
  cost **12 minutes and 10.4 GB** on a 2,000-device × 3,000-test pair.

## 6. Where things are deliberately NOT

This repo is **the deployable product**. Development infrastructure lives one level up in
`/Users/speck/Downloads/docusearch-kickoff/` and is not shipped:

| Not here | Where | Why |
|---|---|---|
| `tests/` | parent workspace | Dev infra, not product. `pytest.testpaths` still says `tests` — run it from the parent. |
| `harness/`, `corpora/`, `redteam/` | parent workspace | Validation assets and adversarial checks; corpora are large and licence-bound. |
| CI (`.github/`) | parent workspace | Gates the workspace, not the wheel. |
| `scripts/` (validation) | parent workspace | This repo's `scripts/` holds only what ships. |
| `docusearch-architecture.md`, `WORKLOG.md`, `CHANGELOG.md`, `RUNBOOK-private-dataset.md` | parent workspace | Process and contract documents. |
| `docusearch.yaml` | untracked (gitignored) | The real config points at a private dataset. The repo ships `docusearch.example.yaml`; pass the real one with `--config`. |

`convert.py` is the one harness-flavoured module that *does* ship — it builds derived test
corpora and lazily imports `reportlab` from the `[dev]` extra, so it costs runtime users
nothing.

## Unverified

Stated here because they could not be confirmed from code alone:

- The **~0.1–0.3 s search / ~0.01–0.6 s report** figures come from the task brief, not from
  a benchmark in this repo. The repo's own measured numbers are the millisecond-scale ones
  in `README.md`; the larger figures presumably include MCP transport and process overhead.
- `install.sh` and `start-server.sh` have **never had a clean-room run** on a fresh machine
  (`BACKLOG.md` §4). Syntax and guard paths are exercised; a real install is not.
- There are **no PowerShell equivalents** (`install.ps1`, `start-server.ps1`) despite Windows
  being a first-class target (`BACKLOG.md` §4).
- `vision_model` is empty on agent-written image descriptions — the `describe_image` tool
  does not record which model looked at the figure (`BACKLOG.md` §3).
